한국에서 ERP나 보험사 시스템을 운영한다면, 단순히 '화면'에 숫자를 띄우는 것으로 끝나지 않습니다. 국세청(NTS), 금융감독원(FSS), 한국거래소(KRX) 등 **규제 기관들이 요구하는 '규격화된 문서'**를 생성하고 전송하는 것이 최종 목적지입니다.

회계 및 법적으로 제출해야 하는 핵심 보고서와 그 규격을 정리해 드릴게요.

---

## 1. 외부 공시용 재무제표 (DART 제출용)
상장사나 대규모 법인은 금융감독원의 **전자공시시스템(DART)**에 보고서를 올려야 합니다.

| 보고서명 | 내용 | 기준 | 제출 형식 |
| :--- | :--- | :--- | :--- |
| **재무상태표 (BS)** | 특정 시점의 자산, 부채, 자본 | IFRS / K-GAAP | **XBRL** (가장 중요) |
| **손익계산서 (IS)** | 특정 기간의 수익과 비용 | IFRS / K-GAAP | **XBRL** |
| **현금흐름표** | 실제 현금의 유입과 유출 | 직접법 / 간접법 | **XBRL** |
| **자본변동표 / 주석** | 자본의 변화 및 상세 설명 | 서술형 포함 | **XBRL** / PDF |

> **💡 개발자 포인트: XBRL이란?**
> **eXtensible Business Reporting Language**의 약자로, 비즈니스 보고용 XML 표준입니다. 단순히 PDF를 올리는 게 아니라, 각 숫자마다 "이건 삼성전자의 2026년 매출액이다"라는 태그를 붙여서 기계가 읽을 수 있게 만든 포맷입니다. C++ 엔진에서 이 XBRL 태그를 자동으로 매핑해주는 모듈이 필요합니다.

---

## 2. 세무 신고용 문서 (국세청 홈택스 제출용)
세금을 내기 위해 국세청에 제출하는 문서들입니다.

| 문서명 | 주요 내용 | 제출 주기 | 형식 |
| :--- | :--- | :--- | :--- |
| **부가가치세 신고서** | 매출/매입 부가세 합산 | 분기/반기 | **Layout(고정길이)** / XML |
| **법인세 신고서** | 기업 이익에 대한 세금 계산 | 연 1회 | Layout / XML |
| **전자세금계산서** | 거래 증빙용 영수증 | 발생 시 수시 | **국세청 표준 XML** |
| **원천징수 이행상황 신고서** | 급여 지급 시 뗀 세금 보고 | 매월 | Layout |

> **💡 개발자 포인트: 전자세금계산서 XML**
> 국세청이 지정한 아주 까다로운 XML 스키마가 있습니다. 공급자의 사업자번호, 금액, 품목 등이 암호화된 전자서명값과 함께 포함되어야 합니다. `Drogon` 서버에서 이 XML을 생성하고 국세청 서버와 **API(또는 중계사업자)**를 통해 통신하게 됩니다.

---

## 3. 보험사 특화 보고서 (금융감독원 제출용)
보험사는 일반 기업보다 훨씬 까다로운 보고 의무가 있습니다.

* **업무보고서:** 보험사의 경영 현황, 자산 운용 내역 등을 담은 수백 페이지 분량의 보고서입니다. (보통 Excel이나 전용 프로그램을 통해 제출)
* **K-ICS (킥스) 리포트:** 보험 부채를 시가로 평가한 결과와 지급여력비율을 보고합니다.
    * **특징:** 수조 원대의 데이터를 시뮬레이션한 결과값이 들어가야 하므로, 사용자님이 만드실 **고성능 엔진**의 결과값이 이 보고서의 기초가 됩니다.

---

## 4. 문서를 만들 때 거치는 '검증 기준'

단순히 텍스트 파일을 만드는 게 아니라, 다음 프로세스를 거쳐야 정식 문서로 인정받습니다.

1.  **데이터 정합성 체크:** 원장(GL)의 숫자와 보고서의 숫자가 1원도 틀리지 않는지 체크.
2.  **스키마 검증 (Schema Validation):** XBRL이나 국세청 XML 규격에 맞는지 XSD(XML Schema Definition) 파일로 사전 검증.
3.  **전자서명 (Digital Signature):** 기업용 공동인증서를 사용하여 문서의 위변조 방지 처리 (PKI 기술 필요).
4.  **전송 로그 관리:** 전송 성공 여부와 수신 확인 번호를 DB에 저장 (나중에 증거로 사용).

## 🏗️ 시스템 설계자로서의 전략

사용자님이 시스템을 설계하신다면, 보고서 기능을 **'별도의 레포팅 마이크로서비스'**로 분리하는 것을 추천합니다.

* **Core 엔진 (C++):** 원장의 데이터를 초고속으로 집계하여 JSON/Protobuf 형태로 전달.
* **Reporting Service (Go/Java):** 전달받은 데이터를 각 기관의 규격(XBRL, 국세청 XML, PDF)으로 변환하고 전자서명을 입혀 전송.

이렇게 분리해야 나중에 국세청 규격이 바뀌거나 새로운 보고서가 추가되어도 **핵심 회계 엔진**을 건드리지 않고 유연하게 대응할 수 있습니다.
