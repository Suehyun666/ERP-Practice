회계 시스템의 코어를 설계하는 것은 마치 **'절대로 무너지지 않는 기초 공사'**를 하는 것과 같습니다. 사용자님이 고성능 시스템(C++, Drogon 등)에 관심이 많으시니, 개발자 관점에서 **"이것만은 절대 놓치지 말아야 할 핵심 설계 포인트"**를 정리해 드릴게요.

## 1. 회계 코어의 '3대 금기' (주의사항)

가장 먼저, 시작할 때 범하기 쉬운 치명적인 실수들입니다.

* **🚫 실수(Floating Point)를 쓰지 마세요:** * `double`이나 `float`로 돈 계산을 하면 0.1 + 0.2가 0.30000000000000004가 되는 대참사가 벌어집니다. 1원이라도 틀리면 결산이 안 됩니다.
    * **해결책:** 무조건 **정수형(Integer)**을 쓰세요. (예: 100원을 `100`으로 저장하거나, 소수점이 필요하면 `10000`으로 저장하고 100으로 나누는 방식). C++라면 `long long`이나 전용 `Decimal` 클래스를 만드세요.
* **🚫 `DELETE`와 `UPDATE`를 봉인하세요:**
    * 회계 데이터는 '기록'입니다. 이미 확정된 전표를 지우거나 수정하는 순간 장부는 오염됩니다.
    * **해결책:** 잘못 썼다면 **역전표(Cancel/Reverse Entry)**를 발행해서 합계를 맞추는 방식(Append-only)만 허용해야 합니다.
* **🚫 시간대(Timezone)의 늪:**
    * 보험사나 증권사는 24시간 돌아갑니다. 서버 시간(KST)으로 기록하면 해외 거래 시 날짜가 꼬입니다.
    * **해결책:** DB 저장은 무조건 **UTC**, 화면 표시는 사용자 지역 시간으로 분리하세요.

## 2. 반드시 만들어야 할 '핵심 엔진'

테이블 외에 코딩으로 구현해야 하는 필수 로직들입니다.

* **차대 밸런스 검증기 (Balance Checker):**
    * 전표가 들어올 때 `차변 합계 == 대변 합계`인지 0.00001초 만에 검증하는 로직입니다.
    * 사용자님이 좋아하는 **Drogon의 비동기 처리**를 활용해, DB에 넣기 직전 메모리 단계에서 광속으로 체크해야 합니다.
* **분개 자동화 엔진 (Posting Engine):**
    * "보험료 결제"라는 비즈니스 언어를 "차) 현금 / 대) 수익"이라는 회계 언어로 바꿔주는 룰 엔진입니다.
    * 이걸 하드코딩하지 말고, **설정 파일(JSON/YAML)**이나 **별도 테이블**로 관리하게 만드세요. 그래야 나중에 보험 상품이 늘어나도 코드 수정 없이 대응됩니다.
* **마감 제어기 (Closing Controller):**
    * 특정 날짜 이전의 데이터는 아예 수정을 시도조차 못 하게 막는 'Lock' 기능입니다.

## 3. 확장 로드맵: 어디서부터 시작할까?

보험사 시스템까지 고려한다면 이 순서가 가장 효율적입니다.

### **[1단계] General Ledger (총계정원장) - "기초 체력"**
* 가장 먼저 **계정과목 마스터**와 **전표 테이블**부터 만드세요.
* "어떤 거래든 일단 전표로 남길 수 있다"는 상태가 되어야 합니다. 이게 없으면 뒤에 뭘 만들어도 사상누각입니다.

### **[2단계] Sub-ledger (보조부) - "근육 붙이기"**
* 이제 회계 전표와 연결될 **보험 계약(Policy)** 테이블, **고객(Customer)** 테이블을 만듭니다.
* 이때 **'연결 고리(Reference ID)'**를 잘 설계해야 합니다. "이 전표는 123번 보험 계약 때문에 생긴 거야"라는 관계를 맺어주는 단계입니다.

### **[3단계] Settlement Engine (정산 엔진) - "지능 부여"**
* 수만 명의 보험료가 들어왔을 때, 이걸 일일이 사람이 전표를 칠 수 없죠?
* **고성능 배치(Batch) 프로세스**를 만듭니다. 밤사이에 수백만 건의 결제 데이터를 읽어서 자동으로 회계 전표 수백만 줄을 생성하는 C++ 엔진이 활약할 차례입니다.

## 💡 사용자님을 위한 '고급 설계 팁'

사용자님이 **분산 시스템**을 좋아하시니, 회계 코어를 만들 때 **'이벤트 소싱(Event Sourcing)'** 개념을 도입해 보세요.

1.  모든 비즈니스 사건(결제, 사고 등)을 **Immutable한 로그**로 먼저 저장합니다.
2.  이 로그를 읽어서 **회계 전표**를 생성하는 '프로젝션(Projection)' 과정을 거칩니다.
3.  이렇게 하면 나중에 회계 규칙이 바뀌어도, 과거 로그를 다시 돌려서(Replay) 과거의 장부를 새로운 규칙으로 재구성할 수도 있습니다. (보험사 IFRS17 대응 시 매우 강력한 무기가 됩니다.)
