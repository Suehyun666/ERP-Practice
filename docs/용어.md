아, SAP의 **'전설적인 테이블'**들을 직접 뜯어보고 싶으시군요!

결론부터 말씀드리면, **SAP라는 프로그램 내부에는 이미 완벽하게 존재하지만, 오픈소스 SQL 파일처럼 외부에서 그냥 다운로드받을 수는 없습니다.** SAP는 폐쇄형 시스템(Proprietary)이라서 그 구조를 데이터베이스 서버(Oracle, HANA 등)에 직접 SQL을 날려 만드는 게 아니라, SAP 전용 개발 도구로 생성하기 때문입니다.

하지만 전 세계 수많은 개발자가 이를 분석해두었기에, **그 '설계도(스키마)'가 어떻게 생겼는지는 명확히 알 수 있습니다.** 사용자님이 자신의 시스템을 만들 때 참고할 수 있도록 SAP의 핵심 스키마를 요약해 드릴게요.

## 1. 전표 헤더: `BKPF` (Accounting Document Header)
전표 한 장의 '공통 정보'입니다.

| 필드명(SAP) | 설명 | 데이터 타입 (추천) | 비고 |
| :--- | :--- | :--- | :--- |
| **MANDT** | 클라이언트 (테넌트 ID) | VARCHAR(3) | 멀티테넌시용 |
| **BUKRS** | 회사 코드 | VARCHAR(4) | 법인 구분 |
| **BELNR** | **전표 번호** | VARCHAR(10) | **PK** |
| **GJAHR** | 회계 연도 | NUMERIC(4) | **PK** |
| BLDAT | 증빙 일자 | DATE | 영수증 날짜 |
| BUDAT | **전기 일자** | DATE | **실제 장부에 반영되는 날짜** |
| TCODE | 트랜잭션 코드 | VARCHAR(20) | 어떤 메뉴에서 만들었나 |

## 2. 전표 상세: `BSEG` (Accounting Document Segment)
전표의 '줄(Line)' 데이터입니다. 한 전표에 여러 줄이 붙습니다.

| 필드명(SAP) | 설명 | 데이터 타입 (추천) | 비고 |
| :--- | :--- | :--- | :--- |
| **BELNR** | 전표 번호 | VARCHAR(10) | **FK (BKPF 연결)** |
| **BUZEI** | **줄 번호** | NUMERIC(3) | **PK (한 전표 내 순번)** |
| KOART | 계정 유형 | CHAR(1) | S(G/L), D(고객), K(공급체) 등 |
| **HKONT** | **G/L 계정 코드** | VARCHAR(10) | **어떤 주머니인가?** |
| **SHKZG** | **차대 구분** | CHAR(1) | **S(차변, +), H(대변, -)** |
| DMBTR | 현지 통화 금액 | DECIMAL(23, 2) | 실제 금액 |

## 3. 계정과목 마스터: `SKA1` (G/L Account Master)
회사가 사용하는 모든 계정의 목록입니다.

| 필드명(SAP) | 설명 | 데이터 타입 (추천) | 비고 |
| :--- | :--- | :--- | :--- |
| **KTOPL** | 계정 차트(CoA) | VARCHAR(4) | 계정 체계 그룹 |
| **SAKNR** | **G/L 계정 번호** | VARCHAR(10) | **PK** |
| KTOKS | 계정 그룹 | VARCHAR(4) | 자산/부채/자본 등 분류 |
| XBILK | 재무제표 계정 여부 | BOOLEAN | 손익계산서용인지 대차대조표용인지 |

## ⚠️ 중요: 2026년 현재의 '진짜' 트렌드 (`ACDOCA`)

사용자님이 보험사나 고성능 시스템을 만드신다면, 위의 `BKPF/BSEG` 방식은 **'옛날 방식'**이라는 점을 아셔야 합니다.

최신 SAP(S/4HANA)에서는 **`ACDOCA` (Universal Journal)**라는 테이블 하나로 모든 것을 통합했습니다.
* **이유:** 헤더(`BKPF`) 따로, 상세(`BSEG`) 따로, 잔액(`FAGLFLEXT`) 따로 관리하니까 데이터 정합성 맞추기가 너무 힘들고 조인이 많아 느려지기 때문입니다.
* **해결책:** 모든 데이터를 하나의 거대한 **와이드 테이블(Wide Table)**에 때려 넣고, 인메모리 DB의 성능으로 밀어버리는 구조입니다.
