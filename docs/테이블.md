회계 코어를 제대로 돌리기 위해서는 **'돈의 흐름(Accounting)'**뿐만 아니라 **'누가 처리했는가(IAM)'**와 **'어디 소속인가(Org)'**에 대한 정보가 유기적으로 연결되어야 합니다.

단일 DB 환경에서 자바(Spring Boot + JPA)로 구현할 때 필요한 **핵심 테이블 12개**를 4가지 그룹으로 나누어 정리해 드릴게요.

---

### 1. 조직 및 인사 테이블 (Identity & Organization)
누가 전표를 입력하고, 어느 부서의 비용으로 처리할지 결정하는 기초가 됩니다.

| 테이블명 | 주요 컬럼 | 설명 |
| :--- | :--- | :--- |
| **`Departments`** | `dept_id`(PK), `dept_name`, `parent_dept_id` | 부서 정보 (비용 센터/Cost Center 역할) |
| **`Users`** | `user_id`(PK), `username`, `password_hash`, `status` | 시스템 접속 계정 정보 |
| **`Employees`** | `emp_id`(PK), `user_id`(FK), `emp_name`, `dept_id`(FK), `position` | 실제 직원 정보 (User와 1:1 또는 N:1 관계) |
| **`Roles_Permissions`** | `role_id`, `permission_name` | **직무 분리(SoD)**를 위한 권한 체계 (예: 전표 입력자 vs 승인자) |

---

### 2. 회계 기준 정보 테이블 (Accounting Master)
장부의 뼈대를 구성하는 기준 데이터입니다.

| 테이블명 | 주요 컬럼 | 설명 |
| :--- | :--- | :--- |
| **`Chart_Of_Accounts`** | `account_code`(PK), `account_name`, `account_type`(자산/부채 등), `parent_code` | 계정과목 마스터 (트리 구조) |
| **`Fiscal_Periods`** | `year`, `month`, `is_closed`, `closed_at`, `closed_by` | 월별 마감 상태 관리 (마감 시 데이터 수정 방어) |
| **`Posting_Rules`** | `event_type`(PK), `debit_account`, `credit_account`, `description` | 비즈니스 이벤트를 전표로 바꾸는 룰 엔진용 |

---

### 3. 전표 및 원장 테이블 (Transaction & Ledger)
실제 회계 데이터가 쌓이는 심장부입니다.

| 테이블명 | 주요 컬럼 | 설명 |
| :--- | :--- | :--- |
| **`Journal_Headers`** | `journal_id`(PK), `post_date`, `status`(DRAFT/POSTED), `created_by`(FK), `dept_id`(FK) | 전표의 메타데이터 (누가, 언제, 어느 부서에서) |
| **`Journal_Lines`** | `line_id`(PK), `journal_id`(FK), `account_code`(FK), `debit_amt`, `credit_amt`, `desc` | **핵심 데이터.** 차변/대변 상세 금액 기록 |
| **`Account_Balances`** | `account_code`(PK), `year_month`(PK), `opening_bal`, `debit_sum`, `credit_sum`, `closing_bal` | 계정별 잔액 스냅샷 (보고서 출력 및 성능 최적화용) |

---

### 4. 로그 및 감사 테이블 (Audit & Log)
보안과 감사를 위해 반드시 필요합니다.

| 테이블명 | 주요 컬럼 | 설명 |
| :--- | :--- | :--- |
| **`Audit_Logs`** | `log_id`(PK), `table_name`, `target_id`, `action`(INSERT/UPDATE 등), `user_id`, `timestamp` | 모든 중요 데이터 변경 이력 (역전표 포함 모든 흔적) |
| **`Access_Logs`** | `access_id`(PK), `user_id`, `ip_address`, `accessed_at`, `endpoint` | 개인정보보호법 대응용 시스템 접속 로그 |

---

### 🏗️ 설계 시 '핵심 연결 고리' (DB 관계)

1.  **전표와 직원:** `Journal_Headers`의 `created_by`는 `Users.user_id`를 참조합니다. 이를 통해 "누가 이 돈을 썼는가?"를 추적합니다.
2.  **전표와 부서:** `Journal_Headers`의 `dept_id`는 `Departments`를 참조합니다. "이 비용은 마케팅팀의 비용인가, 개발팀의 비용인가?"를 구분하여 **부서별 손익계산서**를 뽑을 수 있게 합니다.
3.  **계정과 잔액:** `Journal_Lines`에 데이터가 들어오면 `Account_Balances` 테이블의 해당 월 숫자가 업데이트되어야 합니다.

---

### 💡 자바로 구현할 때의 꿀팁 (JPA 기준)

* **`BigDecimal` 매핑:** DB의 금액 컬럼은 `DECIMAL(18, 2)` 정도로 잡고, 자바 엔티티에서는 반드시 `java.math.BigDecimal`로 매핑하세요.
* **복합키 처리:** `Account_Balances`처럼 (계정코드 + 년월)이 PK인 경우, `@IdClass`나 `@EmbeddedId`를 사용하여 자바에서 관리하기 편하게 만드세요.
* **Envers 활용:** 하이버네이트를 쓰신다면 `Audit_Logs` 테이블을 직접 만드는 대신 **Spring Data Envers** 라이브러리를 쓰면 `@Audited` 어노테이션 하나로 변경 이력을 자동 관리할 수 있습니다.

### **다음 단계로 무엇을 도와드릴까요?**

테이블 구조가 잡혔으니 이제 실제 **엔티티(Entity) 설계**나 **전표를 저장하는 핵심 Service 로직**을 짜볼 수 있습니다.

* **"JPA로 전표 헤더와 상세를 한 번에 저장하는 Entity 연관관계 설정"** 코드를 짜드릴까요?
* 아니면 **"비관적 락을 걸어 `Account_Balances`를 안전하게 업데이트하는 SQL/JPA 로직"**을 보여드릴까요?

원하시는 부분을 말씀해 주시면 바로 코드로 구현해 드릴게요! 부서별 정산 같은 복잡한 로직도 가능합니다.
